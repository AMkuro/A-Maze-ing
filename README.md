# A-Maze-ing
---
## 1. 概要

### 1.1 プロジェクトの目的

本プロジェクトでは、設定ファイルに基づいて迷路を自動生成し、その迷路を解き、課題仕様に沿った形式で出力するとともに、可視化まで行うことを目的とする。

入力では、迷路の幅・高さ、入口座標、出口座標、出力先ファイル名、完全迷路か不完全迷路かといった条件を記述した設定ファイルを受け取る。出力では、各セルの壁情報を 4bit / 16進1文字で表現した迷路データ、入口座標、出口座標、入口から出口までの最短経路を表す NEWS 文字列を含むテキストファイルを生成する。また、生成した迷路とその解法経路をターミナル上で可視化できるようにしている。

この課題で達成することは、迷路生成・迷路検証・最短経路探索・シリアライズ・可視化という一連の処理を、責務ごとに分割した設計のもとで実装し、再利用可能な形でまとめることである。あわせて、設定値の妥当性確認や構造検証を通じて、正しい迷路データのみを扱う仕組みを整えることも目的としている。


### 1.2 このプロジェクトで実装したもの

本プロジェクトでは、設定ファイルの読み込みと検証を行う`ConfigLoader`、迷路を生成する`MazeGenerator`、迷路の最短経路を BFS（幅優先探索）で求める`MazeSolver`、迷路データを課題仕様の出力形式へ変換する`MazeSerializer`、迷路構造の妥当性を確認する`MazeValidator`、そして迷路と経路を可視化する`Visualizer`を実装した。これらを`MazeApp`が統括し、設定読み込みから生成、検証、解法、出力、表示までを一連の流れとして実行する構成にしている。

表示方法としては、ターミナル上で迷路を描画するASCIIベースの可視化を採用している。壁・通路・入口・出口・最短経路を区別して表示できるようにしており、実行中には経路表示の切り替えや配色変更、再生成といった操作も行える。

また、本プロジェクトは単なる一回限りの実装ではなく、再利用可能な構成を意識して設計している。たとえば、`MazeGenerator`は設定に応じた迷路生成機構として独立して利用でき、`MazeSolver`は任意の Maze に対して最短経路を返すモジュールとして再利用できる。`MazeSerializer`も内部表現を課題仕様の外部形式へ変換する独立した部品として扱えるため、今後別の可視化手段や出力方式を追加する際にも流用しやすい構成になっていると言える。

## 2. 実行方法

### 2.1 前提環境

本プロジェクトは Python 3.10 以上を前提としている。

必要な主なツールは以下の通りである。

* Python 3.10 以上
* `venv`
* `make`

依存ライブラリは `requirements.txt` にまとめており、通常は仮想環境内にインストールして使用する。仮想環境はプロジェクト直下の `.venv` を用いる構成とし、Makefile から自動で作成・利用できるようにしている。

### 2.2 インストール

依存関係の導入には`Makefile`を使用する。以下のコマンドで既存の `.venv` を作り直し、必要な依存関係をインストールする。

```
make install
```

このコマンドは、プロジェクト直下に新しい仮想環境を作成し、`requirements.txt`に記載されたパッケージをインストールする。

### 2.3 実行

通常の実行は以下のコマンドで行う。

```
make run
```
デフォルトでは`config.txt`を設定ファイルとして使用する。
別の設定ファイルを使う場合は CONFIG 変数を上書きする。

内部的には`a_maze_ing.py`が呼ばれ、設定ファイルの読み込み、迷路生成、検証、最短経路探索、出力、可視化までを一連の流れとして実行する。

### 2.4 デバッグ

デバッグ実行には`pdb`を利用できる。以下のコマンドでデバッガ付きで起動する。

```
make debug
```

### 2.5 Lint

コード品質の確認には`flake8`と`mypy`を使用する。

通常の lint は以下で実行する。
```
make lint
```
このコマンドでは、`flake8`によるスタイルチェックと、必要なオプション付きの`mypy`による型チェックを行う。

より厳密な型チェックを行いたい場合は、以下を使用する。
```bash
make lint-strict
```
このコマンドでは、`flake8`に加えて`mypy --strict`を実行する。

### 2.6 Clean

一時ファイルやキャッシュを削除するには以下のコマンドを使用する。

```

```
このコマンドでは主に以下の生成物を削除する。

* `__pycache__`

* `.pytest_cache`
* `.ruff_cache`

## 3. 使用方法

### 3.1 基本的な起動方法

基本的な起動は以下のコマンドで行う。

```
make run
```

### 3.2 プログラムの流れ

##### 1. 設定ファイルを読み込む
`ConfigLoader`が`KEY=VALUE`形式の設定ファイルを読み取り、必須キーの有無、値の形式、座標範囲などを検証したうえで`AppConfig`を生成する。
##### 2. 迷路を生成する
`MazeGenerator`が設定値に基づいて迷路を生成する。
迷路はセル単位の内部表現で管理され、各セルは`North / East / South / West`の壁情報を`bit`で保持する。
##### 3. 検証する
`MazeValidator`が生成された迷路について、グリッドサイズ、壁整合性、外周の閉包性、3×3の開放領域の有無などを確認する。
`PERFECT=False`の場合は、不完全迷路になっていることも追加で確認する。
##### 4. 最短経路を求める
`MazeSolver`が BFS（幅優先探索）を用いて入口から出口までの最短経路を探索し、経路座標列とNEWS文字列を生成する。
##### 5. 出力ファイルを書く
`MazeSerializer`が迷路データを課題仕様の 4bit / 16進形式へ変換し、入口座標・出口座標・最短経路を付加して最終的な出力文字列を生成する。
生成された文字列は設定ファイルで指定された出力先ファイルに保存される。
##### 6. 可視化する
`Visualizer`が迷路、入口、出口、最短経路をターミナル上に描画する。
実行中は、迷路の再生成、最短経路表示の切り替え、配色変更などの操作が可能である。

## 4. 設定ファイル形式

### 4.1 形式

設定ファイルは1行につき1つの設定を記述する `KEY=VALUE` 形式で構成する。
行頭または行全体が `#` で始まるものはコメント行として扱い、読み込み時には無視する。また空行も同様に無視する。

各キーと値の間には `=` を用い、前後に空白が含まれていてもパースできるようにしている。
たとえば `ENTRY = 0,0` のような記述も有効である。
デフォルト設定ファイルは、プロジェクト直下の `config.txt` である。

通常実行ではこのファイルを用い、必要に応じて別の設定ファイルを指定できる。
### 4.2 必須キー

| キー | 意味 | 値の形式 |
| --- | --- | --- |
| `WIDTH` | 迷路の幅 | 正の整数 |
| `HEIGHT` | 迷路の高さ | 正の整数 |
| `ENTRY` | 入口座標 | `x,y` 形式の整数ペア |
| `EXIT` | 出口座標 | `x,y` 形式の整数ペア |
| `OUTPUT_FILE` | 出力ファイル名 | 空でない文字列 |
| `PERFECT` | 完全迷路にするかどうか | `True` / `False` |

### 4.3 任意キー

| キー | 意味 | 値の形式 |
| --- | --- | --- |
| `SEED` | 乱数シード | 整数、または省略 |

`SEED` を指定した場合は、同じ設定に対して同じ迷路を再現できる。
`SEED` を省略した場合は、その都度異なる迷路が生成される。

### 4.4 設定例

以下は設定ファイルの一例である。

```text

# normal config

WIDTH=15
HEIGHT=10

# coordinates are written as x,y

ENTRY=0,0
EXIT=14,9
OUTPUT_FILE=maze.txt
PERFECT=True

# optional settings
SEED=42
```

### 4.5 バリデーション条件

* `WIDTH`と`HEIGHT`は、ともに0より大きい整数でなければならない。

* `ENTRY`と`EXIT`は、迷路の範囲内に存在しなければならない。つまり、0 <= x < `WIDTH`かつ 0 <= y < `HEIGHT` を満たす必要がある。

* 入口と出口は同一座標であってはならない。

* `PERFECT`は真偽値として解釈可能な値でなければならない。想定する値は`True / False`である。

* `OUTPUT_FILE`は空文字列であってはならず、改行や NUL 文字など、ファイル名として不適切な文字を含んではならない。

* `SEED`は整数、または未指定でなければならない。

## 5. 迷路の要件

### 5.1 必須要件

本プロジェクトで生成される迷路は、設定ファイルに基づいてランダムに生成される。
ただし、`SEED` が指定された場合は乱数系列を固定し、同じ設定に対して同一の迷路を再現できるようにしている。
`SEED` が未指定の場合は、その都度異なる迷路が生成される。

迷路はセル単位で管理し、各セルは`North / East / South / West`の4方向について壁情報を持つ。

各方向の壁は`bit`で表現され、セルごとに4bitの壁情報として扱う。

入口 `entry` と出口 `exit` は設定ファイルから与えられ、迷路の範囲内に存在しなければならない。
また、入口と出口は同一座標であってはならない。

生成される迷路は、入口から出口へ到達可能であるだけでなく、迷路全体として到達可能なセル群から孤立セルを持たない構造であることを要求する。
すなわち、迷路内部で使用されるセルは互いに接続されており、解法不能な独立領域を作らないことを前提とする。

隣接セル間の壁情報は双方向で整合していなければならない。
たとえば、あるセルの East が閉じているなら、右隣のセルの West も閉じていなければならない。
同様に、South と North の対応についても一致している必要がある。

また、迷路の外周は閉じていなければならない。
すなわち、最上段の North、最下段の South、最左列の West、最右列の East は、それぞれ外部へ開放されないことを要求する。

### 5.2 開放領域の制約

本プロジェクトでは、3×3の開放領域を生成しないことを制約としている。
ここでいう3×3の開放領域とは、3行3列のセルブロックにおいて、内部の横方向および縦方向の境界がすべて開いている状態を指す。
この制約により、単なる広い空白領域のような構造を避け、迷路としての形状と探索性を保つことを意図している。

### 5.3 “42” パターン

本プロジェクトでは、迷路中央付近に “42” パターンを埋め込む仕組みを実装している。

このパターンは特別な保護セルとして扱われ、通常の通路掘削処理の対象から除外されるため、迷路生成の過程でこのパターンが破壊されないようにしている。

ただし、迷路サイズが小さすぎる場合には、中央に十分な大きさで “42” パターンを配置できない。
その場合は無理に埋め込まず、通常の迷路生成のみを行う。

### 5.4 PERFECT=True の意味

`PERFECT=True` の場合、生成される迷路は「完全迷路」である。

「完全迷路」とは、すべてのセルが連結であり、かつループを持たない迷路を指す。
この性質により、任意の2セル間の経路は一意に定まり、特に entry から exit への経路もただ1つになる。

一方で`PERFECT=False` の場合は、完全迷路を生成した後に追加で壁を開放することで少なくとも1つのループを作り、不完全迷路とする。
その際も3×3の完全開放領域を作らないこと、つまり迷路全体の整合性を保つことを条件としている。

## 6. 出力ファイル形式

### 6.1 基本形式

出力ファイルは、迷路本体を表すグリッド部と、追加情報を表すフッター部から構成される。

グリッド部では、各セルを 16進数1文字で表現する。1行は迷路の 1行分のセル列に対応し、各行の文字数は迷路の幅と一致する。

グリッド部の後には1行の空行を入れ、その後に入口座標、出口座標、最短経路文字列をそれぞれ1行ずつ出力する。

また、すべての行は改行文字 `\n` で終わる形式とする。

### 6.2 壁のビット表現

| ビット | 方向 |
| --- | --- |
| 0 | North |
| 1 | East |
| 2 | South |
| 3 | West |

各セルは4bitで壁情報を持ち、この4bitをそのまま 16進数1文字に変換して出力する。

### 6.3 エンコードの意味

各ビットは、その方向の壁が閉じているかどうかを表す。
閉じた壁は `1`、開いた壁は `0` とする。

たとえば、
* `3`（2進数 `0011`）は、North と East の壁が閉じており、South と West の壁が開いていることを表す。

* `A`（2進数 `1010`）は、East と West の壁が閉じており、North と South の壁が開いていることを表す。

* `F`（2進数 `1111`）は、4 方向すべての壁が閉じていることを表す。

* `0`（2進数 `0000`）は、4 方向すべてが開いていることを表す。

内部実装では追加の管理ビットを持つ場合があるが、出力時には下位4bitのみを使用し、課題仕様の16進表現へ変換する。

### 6.4 空行の後に書く情報

空行の後には、以下の3つの情報をこの順で1行ずつ出力する。

1. 入口座標
2. 出口座標
3. 入口から出口までの最短経路を表す NEWS 文字列

入口座標と出口座標は、設定および課題仕様で定められた座標表現に従って出力する。

### 6.5 最短経路の表現

最短経路は、入口から出口までの移動方向を `N`、`E`、`S`、`W` の 4 文字で表現した文字列として出力する。

各文字の意味は次の通りである。

* `N`: 1 セル上へ移動
* `E`: 1 セル右へ移動
* `S`: 1 セル下へ移動
* `W`: 1 セル左へ移動

この文字列は、`MazeSolver`が BFS により求めた最短経路をもとに生成される。

### 6.6 出力例

以下は出力形式の一例である。

```

95153
C3A9F
AC6C2

0,0
4,2

EESSWN
```

## 7. ビジュアライザ

### 7.1 採用した表示方式

本プロジェクトでは、可視化方式として ASCII ベースのターミナル表示を採用した。

この方式を選んだ理由は、追加の GUI 環境に依存せず、標準的なターミナル環境でそのまま動作させやすいからである。

また、迷路生成・解法・出力といった本課題の中心的な処理と分離しやすく、描画部分を独立した責務として整理しやすい点も利点である。

表示には Unicode のブロック文字と ANSI エスケープシーケンスを用いており、壁・通路・入口・出口・最短経路を視覚的に区別できるようにしている。

### 7.2 表示する要素

ビジュアライザでは、迷路を構成する以下の要素を表示する。

* ##### 壁
  各セルが持つ`North / East / South / West`の壁情報をもとに、迷路の輪郭と内部構造を描画する。

* ##### entry
  入口セルを他の通路と区別して表示する。

* ##### exit
  出口セルを他の通路と区別して表示する。

* ##### 解法経路
  `MazeSolver`が求めた最短経路をハイライト表示し、入口から出口までの経路を視覚的に追えるようにする。
  ただし、経路表示は常時固定ではなく、ユーザー操作に応じて表示・非表示を切り替えることが可能である。

### 7.3 ユーザー操作

実行中のビジュアライザでは、ターミナルメニューを通じて以下の操作を行える。

* ##### 再生成

  新しい迷路を再度生成して表示する。
  `SEED` が固定されていない場合は、毎回異なる迷路が再生成される。

* ##### 最短経路の表示切替
  入口から出口までの最短経路を表示または非表示に切り替える。

* ##### 壁色の変更
  壁・通路・経路・入口・出口に対する配色を変更し、表示の見やすさや印象を調整できる。

### 7.4 ビジュアライザ設計

ビジュアライザは、描画対象として `Maze` と `Solution` を受け取る。
`Maze` からは迷路の幅・高さ・壁情報・入口・出口を参照し、`Solution` からは最短経路の座標列を受け取る。

内部表現として迷路はセル単位で管理されており、各セルは4bitの壁情報を持つ。

ビジュアライザはこのセルベースの表現を、そのまま直接描画するのではなく、一度描画用バッファに展開してから文字グリッドへ変換している。
これにより、壁を表す線やブロック文字を整った形で表示できる。

また、描画時には入口・出口・経路の強調表示を別レイヤとして扱い、壁情報そのものとは独立して色付けを行っている。
この構造により、迷路本体の情報と、経路や注目セルの視覚的強調とを分離して管理できるようにしている。

## 8. 採用アルゴリズム

### 8.1 採用したアルゴリズム

本プロジェクトでは、迷路生成アルゴリズムとして、スタックを用いた反復的な DFS（深さ優先探索）による Recursive Backtracker を採用した。

### 8.2 アルゴリズムの概要

迷路生成は、すべてのセルを壁で閉じた状態から開始し、未訪問セルを深さ優先的にたどりながら通路を開通させていくことで行う。
現在セルから進める未訪問の隣接セルがある場合は、そのうち1つをランダムに選び、両セル間の壁を取り除いて移動する。
未訪問の隣接セルが存在しない場合は、スタックを用いて1つ前のセルへ戻り、別の分岐を探索する。
この操作を、到達可能な全セルを訪問し終えるまで繰り返すことで、ループのない完全迷路を生成する。

### 8.3 このアルゴリズムを選んだ理由

* 実装が比較的単純であり、迷路生成の流れを明確に追いやすい。
* 完全迷路を自然に生成でき、`PERFECT=True` の要件と整合を取りやすい。
* 再帰ではなくスタックを使うことで、Python における再帰深度制限を避けつつ、安定した実装にできる。
### 8.4 利点

* 完全迷路を簡潔に生成できる。
* 深く掘り進める性質があるため、迷路らしい長い通路が現れやすい。
* 実装量が比較的少なく、シンプルなアルゴリズムである。

### 8.5 欠点・制約

* 隣接セルの選択自体はランダムであるが、アルゴリズム全体としては深さ優先に進むため、長い通路や局所的な分岐が生じる可能性がある。

### 8.6 他案との比較

* ##### recursive backtracker との比較
   本プロジェクトで採用した方法は、理論的には Recursive Backtracker そのものである。ただし、再帰関数による実装ではなく、スタックを用いた反復的な DFS として実装している。そのため、アルゴリズム上の性質は維持しつつ、Python の再帰深度制限を回避できる点が実装上の利点である。
* ##### randomized Prim との比較
   Randomized Prim は、前線にある候補壁の中から1つを選びながら領域を拡張していくため、局所的な分岐が多く、比較的均質な迷路になりやすい。
* ##### randomized Kruskal との比較
   Randomized Kruskal は、各セルを独立集合として扱い、壁をランダムに除去しながら連結成分を統合していく方法である。この手法では Union-Find などの追加データ構造が必要となり、実装がやや重くなる

## 9. 再利用モジュールの使い方

### 9.1 基本的な利用例


最小構成では、設定を読み込み、迷路を生成し、最短経路を求め、必要に応じてシリアライズや可視化を行う。

```python
from mazegen.ConfigLoader import ConfigLoader
from mazegen.MazeGenerator import MazeGenerator
from mazegen.MazeSolver import MazeSolver
from mazegen.MazeSerializer import MazeSerializer

config = ConfigLoader().load("config.txt")
maze = MazeGenerator.generate(config)
solution = MazeSolver.solve(maze)
output = MazeSerializer.serialize(maze, solution)

print(output, end="")
```

設定ファイルを介さず、直接パラメータを与えて利用する構成へ拡張することも可能である。

### 9.2 指定できる引数

迷路生成に関連する主なパラメータは以下の通りである。

* ##### width
    迷路の幅を表す正の整数。
* ##### height
    迷路の高さを表す正の整数。
* ##### entry
    入口座標。x, y 形式で指定する。
* ##### exit
    出口座標。x, y 形式で指定する。
* ##### perfect
    完全迷路を生成するかどうかを表す真偽値。`True`の場合はループを持たない完全迷路、`False`の場合は少なくとも 1つのループを持つ不完全迷路を生成する。
* ##### seed
    指定した場合は再現可能な迷路生成ができる。

### 9.3 取得できる情報

生成および解法後には、次のような情報を取得できる。

* ##### 生成済み迷路
    `Maze`オブジェクトとして取得できる。
* ##### 壁情報
    `maze.grid`を通じて各セルの壁情報にアクセスできる。
* ##### entry / exit
    `maze.entry`, `maze.exit`として取得できる。
* ##### 迷路サイズ
    `maze.width`, `maze.height`として取得できる。
* ##### 解法結果
    `Solution`オブジェクトとして取得できる。
* ##### 経路座標列
    `solution.path`により、入口から出口までの最短経路を座標列として参照できる。
* ##### NEWS 文字列
    `solution.news`により、最短経路を`N / E / S / W`の文字列として取得できる。

### 9.4 パッケージ化

本プロジェクトはパッケージ化して利用できるように構成している。
ビルドには`PMakefile`を利用し、以下のコマンドで生成できる。
```
make build-pkg
```
このコマンドにより、プロジェクト直下に配布物が生成される。
想定される配布形式は以下である。

* ##### *.whl
    Python パッケージとしてインストールしやすい wheel 形式。
* ##### *.tar.gz
    ソース配布用のアーカイブ形式。

生成した wheel を仮想環境へインストールする場合は、以下を利用できる。
```
make install-pkg
```
これにより、生成済みの`mazegen-*.whl`を仮想環境にインストールできる。
パッケージ化しておくことで、アプリケーションとしてだけでなく、迷路生成・解法ライブラリとして別環境から利用しやすくなる。

### 10.2 各ファイル / ディレクトリの役割

* ##### `a_maze_ing.py`
  設定ファイルを受け取り、迷路生成から検証、解法、出力、可視化までの一連の流れを起動する。

* ##### `config.txt`
  迷路生成に必要な設定を記述する設定ファイルである。幅、高さ、入口、出口、出力ファイル名、完全迷路かどうかなどを指定する。

* ##### `mazegen/`
  プロジェクト本体のソースコードを配置したパッケージディレクトリである。迷路生成、検証、解法、シリアライズ、可視化などの各責務をモジュールごとに分離している。

* ##### `mazegen/ConfigLoader.py`
  設定ファイルを読み込み、各値を検証し、AppConfig として扱える形に変換する。

* ##### `mazegen/MazeApp.py`
  アプリケーション全体の進行を統括する。設定読み込み、迷路生成、検証、解法、出力、可視化を順に呼び出す。

* ##### `mazegen/MazeGenerator.py`
  迷路生成を担当する。セル単位の壁情報を持つ内部表現を構築し、必要に応じて完全迷路または不完全迷路を生成する。

* ##### `mazegen/MazeSolver.py`
  BFS により入口から出口までの最短経路を求める。経路座標列と NEWS 文字列を返す。

* ##### `mazegen/MazeSerializer.py`
  迷路内部表現を課題仕様の 4bit / 16進形式へ変換し、入口座標・出口座標・最短経路を含む最終出力文字列を生成する。

* ##### `mazegen/MazeValidator.py`
  生成された迷路が仕様を満たしているか検証する。グリッドサイズ、壁整合性、外周、3×3 開放領域、不完全迷路条件などを確認する。

* ##### `mazegen/MazeVisualizer.py`
  迷路と最短経路をターミナル上に描画する。入口、出口、経路、壁を色付きで表示し、表示切替や色変更にも対応する。

* ##### `\README.md`
  プロジェクトの概要、実行方法、設計、出力形式、アルゴリズム、再利用方法などを説明する文書である。

* ##### `Makefile`
  仮想環境の構築、依存関係のインストール、実行、デバッグ、lint、clean といった操作をまとめた補助ファイルである。

* ##### `requirements.txt`
  実行および開発に必要な Python パッケージを列挙する。

* ##### `pyproject.toml`
  パッケージングやビルド設定を管理する設定ファイルである。
